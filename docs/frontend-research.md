# 前端和博客案例调研

## 博客信息架构

成熟博客通常不是只靠标签，而是同时使用：

- 分类：一篇文章通常只有一个主分类，用来表达内容栏目。
- 标签：一篇文章可以有多个标签，用来表达横向主题。
- 归档：按年份、月份或时间线浏览。
- 搜索：按标题、摘要、正文、标签检索。
- 相关文章：根据分类和标签推荐。

对 llm-selfwiki 来说，建议这样设计：

- `category`：内容的主归属，例如「日记」「技术」「学习」「项目」「观点」「公开记忆」。
- `tags`：更细的主题，例如「self-LLM」「RAG」「Next.js」「复盘」。
- `visibility`：私有、草稿、公开，决定是否进入公开博客和 self-LLM。
- `type`：系统内部内容类型，例如 entry、note、post、memory。

`type` 负责系统工作流，`category` 负责公开阅读体验，两者不要混用。

## 组件库方向

建议优先考虑：

- Radix UI：无样式、可访问性好，适合做底层交互组件。
- shadcn/ui：基于 Radix 和 Tailwind 的组件集合，代码可复制进项目，适合可控定制。
- Ariakit：无样式、可访问性强，适合复杂交互。
- React Aria：能力强，但抽象偏重。

暂不建议一开始使用：

- Ant Design：后台能力强，但视觉风格重，个人博客和 self-LLM 会显得企业后台化。
- Material UI：成熟，但默认风格辨识度太强，定制成本不低。
- Chakra UI：开发体验好，但长期视觉一致性和深度定制不如 shadcn/Radix 直接。

## 当前项目建议

第一阶段先不引入完整 UI 框架，继续使用手写 CSS，原因：

- 当前页面少，手写样式更容易控制产品气质。
- 还在探索 self-LLM 的交互形态，过早套组件库会限制表达。
- 后续需要弹窗、下拉菜单、标签选择器、命令面板时，再引入 Radix 或 shadcn。

适合优先做的组件：

- 分类/标签筛选栏
- 文章列表卡片
- self-LLM 引用来源卡片
- 工作台内容编辑器
- 图片上传预览
- 登录表单
- 命令面板

## 视觉方向

不要做普通博客模板。这个项目更像「个人知识操作系统 + 公开 AI 人格接口」。

建议视觉关键词：

- 安静
- 工具感
- 知识密度
- 可阅读
- 有一点未来感，但不要赛博装饰
- 公开页面更像杂志/知识库
- 私有页面更像工作台
