# llm-selfwiki 产品讨论稿

## 定位

llm-selfwiki 是一个个人动态博客、思绪处理系统和 self-LLM 个人 AI 分身。它运行在自己的服务器上，主要服务于私有写作、复盘、学习规划和知识沉淀，同时基于公开内容和可公开的人格设定，提供一个面向访客的 AI 对话服务。

它不是单纯的博客系统。核心目标是把每天散乱的想法，逐步变成可回顾、可连接、可发布、可继续生长的个人知识资产，并最终支撑一个能够模拟个人表达方式、知识结构和思考偏好的 self-LLM。

## 要解决的问题

1. 思绪太散，很多想法没有真正落地。
2. 学习缺少稳定的计划、执行、复盘闭环。
3. 日记、笔记、博客、知识库之间割裂。
4. LLM 辅助复盘有价值，但桌面 agent 的使用场景受限。
5. 公开写作可以先由动态服务提供，后续如果需要，再增加静态部署能力。
6. 普通个人博客只能展示静态文本，无法让访客以对话方式理解一个人的观点、知识和思考方式。

## 一句话定义

新时代的个人动态博客：默认私有、LLM 辅助、知识库驱动，并且提供一个基于个人公开知识和人格设定的 self-LLM。

## 第一版 MVP 范围

第一版要足够小，能稳定跑在 2 核 2G 的服务器上。

### 1. 思绪收集箱

- 快速记录每天的杂乱想法
- 支持日记式输入
- 支持 Markdown 或富文本
- 支持图片、截图、附件
- 支持标签和简单状态
- 默认私有

### 2. 复盘系统

- 每日复盘页面
- 每周复盘页面
- 从原始想法中整理出任务、笔记、博客草稿
- 使用 LLM 辅助总结和反思

### 3. 个人知识库

- Markdown 笔记
- 笔记之间可以互相引用
- 标签/主题索引
- 搜索
- 图谱视图可以后置，不放进第一版核心

### 4. 博客发布

- 指定某些笔记或文章可以公开
- 可以先由动态服务直接提供公开页面
- 后续可选导出为静态站点
- 静态导出时，可以定时或手动提交到 GitHub，并通过 Cloudflare Pages 部署

### 5. LLM Wiki 集成

- 用 LLM 总结、分类、连接笔记
- 从长期记录中生成 wiki 风格页面
- LLM 配置只放在服务端
- 第一版避免做复杂的常驻 agent

### 6. self-LLM 个人 AI 分身

- 面向公开访客提供对话入口
- 回答基于公开文章、公开笔记、公开知识库和明确允许公开的人格设定
- 尽量模拟个人表达方式、价值偏好和思考习惯
- 对私密日记、私有复盘、未公开笔记保持隔离
- 当公开资料不足时，应明确表示不知道，而不是编造个人经历

## 推荐架构

### 私有运行服务

用一个轻量的全栈 Web 应用承载私有功能：

- Web 框架：SvelteKit、Next.js 或 Nuxt
- 数据库：SQLite
- 内容格式：数据库记录内容和元数据，图片/附件作为媒体资产单独管理
- LLM 调用：服务端接口或后台任务
- self-LLM：检索公开知识后再生成回答
- 定时任务：cron 或 systemd timer

对于 2 核 2G 服务器，第一版不建议拆成多个服务。

### 公开博客

公开博客可以有两种模式。

第一阶段可以直接由动态服务提供：

- 使用同一个 Next.js 服务
- 根据内容可见性展示公开文章
- 不需要额外导出流程
- 实现简单，适合早期验证

后续如果需要更强的部署隔离，可以增加静态输出：

- 只包含公开内容
- 不依赖私有 API
- 不暴露私有数据库
- 可以部署到 Cloudflare Pages
- 可以由 GitHub 触发部署

这样早期不会过度设计，后期也保留静态化能力。

### self-LLM 公开服务

self-LLM 不应该直接读取全部私有数据库。它需要一个明确的可公开知识边界：

- 公开文章
- 公开笔记
- 公开知识库页面
- 人格设定文件
- 可公开的经历、项目、观点摘要

对话流程建议是：

1. 用户提问。
2. 系统检索公开知识库。
3. 拼接人格设定、相关资料和回答规则。
4. LLM 生成回答。
5. 返回时可以附带引用来源。

这样它不是简单的通用聊天机器人，而是一个带有个人知识和人格约束的公开 AI 服务。

## 数据模型草案

### Entry：原始记录

用于保存每天的原始输入。

- id
- 标题
- 正文内容
- 创建时间
- 更新时间
- 心情/精力，可选
- 标签
- 可见性：私有/公开/草稿
- 来源：日记/收集箱/手动导入
- 关联媒体资产

### Note：知识库笔记

用于保存沉淀后的知识内容。

- id
- 标题
- slug
- 正文内容
- 标签
- 链接关系
- 可见性
- 关联媒体资产

### Task：任务

用于保存从复盘或手动输入中得到的行动项。

- id
- 标题
- 状态
- 截止日期，可选
- 来源 Entry，可选

### Review：复盘

用于保存每日、每周或每月复盘。

- id
- 类型：daily/weekly/monthly
- 时间周期
- 总结
- 决策
- 下一步行动

### Post：公开文章

用于保存最终面向公开博客的文章。

- id
- 标题
- slug
- 正文内容
- 状态：草稿/已发布/归档
- 发布时间
- 来源笔记或原始记录，可选
- 关联媒体资产

### Persona：人格设定

用于描述 self-LLM 应该如何模拟个人表达方式和思考偏好。

- id
- 名称
- 简介
- 语气风格
- 价值偏好
- 专长领域
- 禁止回答的边界
- 更新时间

### Memory：可公开记忆

用于保存 self-LLM 可以使用的公开背景信息。

- id
- 标题
- 内容
- 类型：经历/项目/观点/偏好/常见问答
- 可见性
- 置信度或人工确认状态

### ChatSession：公开对话

用于保存访客和 self-LLM 的对话记录，可按隐私策略决定是否持久化。

- id
- 会话时间
- 用户问题
- 检索到的资料
- 模型回答
- 是否命中边界规则

### Asset：媒体资产

用于保存图片、截图、附件等非文本内容的索引。

- id
- 文件名
- MIME 类型
- 文件大小
- 存储路径
- 创建时间
- 来源对象：Entry/Note/Post，可选

## 推荐的 MVP 使用流程

1. 打开私有 Web 应用。
2. 把当天杂乱想法写进今日收集箱。
3. 晚上进入每日复盘。
4. LLM 生成建议：
   - 今日总结
   - 重要想法
   - 任务
   - 可能沉淀成知识库的笔记
   - 可能发展成博客的草稿
   - 可能加入 self-LLM 的公开记忆
5. 用户手动确认和编辑。
6. 选择部分内容发布。
7. 公开内容先由动态服务直接展示。
8. self-LLM 可以基于已公开内容回答访客问题。
9. 后续如果需要，再增加静态导出和 Cloudflare Pages 部署。

## 实现阶段

### 阶段 0：决策

- 确定框架
- 确定存储方式
- 确定公开博客是否和私有系统放在同一个仓库
- 确定登录方式

### 阶段 1：本地 MVP

- 私有 Web 应用
- SQLite 持久化
- Markdown 编辑器
- 日记/收集箱页面
- 笔记列表和详情页
- 基础搜索

### 阶段 2：复盘系统

- 每日复盘
- 每周复盘
- 任务提取
- LLM 总结接口
- Prompt 模板

### 阶段 3：self-LLM 公开服务

- 人格设定管理
- 公开知识检索
- 公开对话入口
- 回答引用来源
- 私有内容隔离规则

### 阶段 4：静态发布

这一阶段是可选增强，不是第一版必须内容。

- 公开文章模型
- 静态导出
- GitHub 推送脚本
- Cloudflare Pages 部署配置

### 阶段 5：知识扩展

- 双向链接
- 自动生成 wiki 页面
- 主题聚类
- 导入/导出
- 如果仍然有价值，再做图谱视图

## 当前需要讨论的问题

1. 你更想用 TypeScript 技术栈，还是 Python 技术栈？
2. 私有内容应该更像 Web App 那样统一管理，还是更像 Obsidian 那样以文件为中心？
3. 这个系统只给你一个人用，还是未来可能支持多人？
4. 公开博客先由动态服务提供，还是一开始就规划静态导出？
5. 第一版接哪个 LLM：OpenAI 兼容 API、本地模型，还是先抽象接口？
6. 是否必须适配手机写作？
7. 访问方式是密码登录、Tailscale/内网访问，还是两者都要？
8. self-LLM 只能使用公开内容，还是允许使用部分人工确认过的半公开记忆？
9. self-LLM 应该更像“本人语气的知识问答”，还是更像“模拟本人进行观点表达”？
